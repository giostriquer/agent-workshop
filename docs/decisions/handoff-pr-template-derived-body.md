# Decision: derive the `handoff-pr` body from the repo's own PR template

**Date:** 2026-06-18

## Status

Implemented (2026-06-18). `validate-native-plugin.ps1` passes.

## Context

`handoff-pr` packaged a finished branch into a PR artifact whose body it
**invented**: a fixed Summary / Ticket / Validation / Review / Caveats structure.
Most repos that care about PRs already ship a `pull_request_template.md` declaring
the sections and checklist they want. Pasting a parallel, skill-invented structure
into such a repo produces a PR that doesn't match the team's expected shape — the
checklist the repo asks for is missing, and the sections the repo *didn't* ask for
are present.

A second problem the old single-body design carried: it mixed reviewer-facing
content (summary, ticket) with handoff bookkeeping (what was validated this
session, whether a `handoff-review` pass ran, the `gh pr create` command). Pasted
verbatim into a public PR, the bookkeeping reads as noise and can expose internal
scratch paths (e.g. a `tmp/handoff-review-*.md` link).

## The change

1. **Template detection before assembly.** A new step searches for a PR template
   GitHub would honor, case-insensitively: `.github/pull_request_template.md` and
   `.github/PULL_REQUEST_TEMPLATE.md`; any file under `.github/PULL_REQUEST_TEMPLATE/`;
   and the same names (plus the directory form) in the repo **root** and under
   **docs/**.
2. **Body = template, filled verbatim.** When a template exists, the PR body *is*
   that template with its headings, order, checkbox items, and `<!-- comment
   markers -->` preserved. Our content maps into the template's existing fields;
   no section is added, dropped, or renamed. Checklist items are ticked `[x]` only
   when actually verified.
3. **Multiple templates → pick by intent.** If a repo ships a default plus
   `hotfix` / `release` variants, choose by branch name / change intent and record
   which one was used.
4. **No template → built-in fallback**, now trimmed to the reviewer-facing
   Summary / Ticket / Caveats.
5. **Two visibly separate blocks.** The paste-ready **PR body** is the only part
   that goes in the PR description. A separate **handoff notes** block carries the
   opener-only fields — which template was chosen and why, validation provenance,
   review status, and the `gh pr create` command — and is explicitly *not* for the
   PR.
6. **Tooling-agnostic body.** A rule forbids naming editors, bots, or AI
   assistants in the body (no "generated by" footers): describe the change, not how
   it was produced.

## The one judgment call

The previous fallback body carried a **Validation** and a **Review** section. The
new design treats validation *provenance* and review *status* as opener-only and
moves them out of the no-template body entirely. This is a deliberate behavior
change, not an oversight: the request named both as process fields that must not
leak into the public PR description. When a repo template *does* have a
testing/QA field or checklist, validation evidence still maps into it — because
the repo asked for it in its own public body — while the full provenance always
stays in the handoff notes.

## Why this is the right shape for the scaffold

- **More portable.** The skill now bends to the adopting repo's conventions
  instead of imposing its own. That is the scaffold's whole maintenance stance:
  fitness for adoption.
- **More honest.** The public PR carries only what a reviewer needs; the
  handoff bookkeeping travels with the opener, where it belongs.
- **Still tooling-agnostic.** No editor, bot, or assistant is named anywhere in
  the body, so the artifact reads the same regardless of who opens the PR.
